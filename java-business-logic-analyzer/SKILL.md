# Java 业务逻辑分析器

## 分析流程

### 1. 定位分析范围

根据用户指定的模块、接口或类，确定分析边界：

- 若用户指定了具体接口/方法：从该方法入手，向下追踪完整调用链
- 若用户指定了模块/包名：扫描该包下所有 Controller，列出接口清单供用户选择
- 若用户不确定：先用 Glob 扫描 `**/controller/**/*.java` 和 `**/Controller.java`，列出可用接口

### 2. 代码探索策略

按以下顺序逐层深入，使用 Grep 和 Read 工具：

```
Controller → Service → 领域模型/Manager → Repository/Mapper
```

**关键搜索模式：**

- 入口：`@RequestMapping`, `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`
- 服务层：`@Service`, `@Transactional`, `@Component`
- 业务规则：`if/else`, `switch`, `throw`, `Assert`, `Preconditions.check`, 自定义校验注解
- 状态流转：`enum` + `Status/State`, 状态机相关类
- 异常处理：`@ExceptionHandler`, `BusinessException`, 自定义异常类
- 事件/消息：`@EventListener`, `ApplicationEvent`, `@RabbitListener`, `@KafkaListener`

### 3. 业务规则提取要点

分析代码时重点提取：

**a) 业务校验规则**
- 参数校验（`@Valid`, `@NotNull`, 自定义Validator）
- 业务前置条件（权限检查、状态检查、数据存在性检查）
- 业务约束（金额范围、时间窗口、数量限制等）

**b) 核心业务流程**
- 主流程：正常执行路径
- 分支流程：条件判断导致的不同处理路径
- 异常流程：业务异常和系统异常的处理方式

**c) 状态流转**
- 识别枚举中的状态定义
- 追踪状态变更的触发条件和目标状态
- 记录不合法的状态转换（被代码阻止的）

**d) 数据变更**
- 哪些数据被创建/更新/删除
- 数据变更的前置条件
- 关联数据的级联操作

**e) 外部交互**
- 调用了哪些外部服务（RPC/HTTP/MQ）
- 发送了哪些事件或消息
- 缓存的读写策略

### 4. 输出格式

按以下 Markdown 模板组织输出，根据实际分析内容裁剪章节：

```markdown
# [模块/接口名称] 业务逻辑分析

## 概述
- **所属模块**：xxx
- **接口路径**：`POST /api/xxx`
- **核心职责**：一句话描述

## 业务流程

### 主流程
1. 步骤1：描述（`ClassName:lineNumber`）
2. 步骤2：描述（`ClassName:lineNumber`）
...

### 分支流程
- **条件A**：当 xxx 时 → 执行 yyy
- **条件B**：当 xxx 时 → 执行 zzz

## 业务规则

| # | 规则描述 | 触发条件 | 处理方式 | 代码位置 |
|---|---------|---------|---------|---------|
| 1 | xxx | xxx | 抛出异常/返回错误 | `Class:line` |

## 状态流转（如适用）

| 当前状态 | 触发动作 | 目标状态 | 前置条件 |
|---------|---------|---------|---------|
| CREATED | 提交审核 | PENDING | 数据完整 |

## 数据变更

| 操作 | 目标表/实体 | 关键字段 | 条件 |
|-----|-----------|---------|------|
| INSERT | Order | amount, status | 校验通过后 |

## 外部依赖
- **调用 xxx 服务**：用途说明
- **发送 xxx 消息**：触发条件

## 异常处理

| 异常类型 | 触发条件 | 处理方式 |
|---------|---------|---------|
| BusinessException | xxx | 返回错误码 |

## 关键代码引用
- 入口：`XxxController.java:42`
- 核心逻辑：`XxxService.java:85`
- 数据访问：`XxxMapper.java:20`
```

### 5. 分析原则

- **先全局后局部**：先理解模块职责，再深入方法细节
- **追踪数据流**：关注入参如何被处理、转换、持久化
- **识别隐含规则**：注释、命名、常量定义中隐含的业务语义
- **标注不确定项**：对于无法仅从代码确定的业务含义，明确标注"需确认"
- **引用代码位置**：所有业务规则和关键步骤都附带 `文件:行号` 引用

## 多模块分析

当需要分析整个模块或多个相关接口时：

1. 先生成接口清单概览（路径、方法、简要描述）
2. 询问用户优先分析哪些接口
3. 逐个分析，最后生成模块级汇总文档

详细的分析模式和高级用法参见 [references/analysis-patterns.md](references/analysis-patterns.md)。
