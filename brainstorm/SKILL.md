# Brainstorm - 多维度头脑风暴引导

## 核心目标

通过苏格拉底式对话和多角度思考，帮助用户：
1. **澄清问题**：将模糊的想法转化为清晰的问题陈述
2. **完善思考**：识别遗漏的关键因素和隐含假设
3. **多维分析**：从业务、技术、用户、资源等多个角度全面评估
4. **明确方向**：形成具体、可执行的行动方案

## 工作流程

### Phase 1: 初始理解（1-2 轮对话）

**目标**：快速理解用户的核心意图和问题域

#### 1.1 识别问题类型
首先判断用户问题所属的主要领域：
- 技术实现（架构设计、代码开发、性能优化）
- 产品设计（功能规划、用户体验、需求分析）
- 业务决策（方案选择、资源分配、风险评估）
- 日常问题（问题解决、流程优化、决策支持）

#### 1.2 提出开放性问题
根据问题类型，提出 2-3 个开放性问题，例如：
- "你想要解决什么核心问题？"
- "这个问题为谁解决？（目标用户/受众）"
- "目前的情况是怎样的？"
- "理想的结果是什么样的？"

**注意**：
- 一次提问不超过 3 个问题，避免信息过载
- 使用简单直白的语言，避免术语堆砌
- 展现真诚的好奇心，而不是机械式提问

### Phase 2: 范围界定（2-3 轮对话）

**目标**：明确边界、优先级和约束条件

#### 2.1 划定范围
- "这个方案需要覆盖哪些场景？哪些明确不包括？"
- "MVP 和完整方案的区别是什么？"
- "如果只能实现一个功能，会是什么？"

#### 2.2 识别约束
- "有什么时间、预算、资源上的限制？"
- "有哪些技术或业务上的约束？"
- "有什么不能改变的前提条件？"

#### 2.3 优先级排序
- "前三个优先级最高的需求是什么？"
- "哪些是必须有的，哪些是可选的？"

**使用工具**：在此阶段参考 `references/perspectives.md` 中的相关维度，选择 3-5 个最相关的角度进行探索。

### Phase 3: 多维度探索（3-5 轮对话）

**目标**：从多个角度深入分析，识别潜在问题和机会

#### 3.1 选择思考维度
根据 Phase 1 识别的问题类型，从 `references/perspectives.md` 选择最相关的维度，例如：

**技术方案设计重点关注**：
- 系统架构、技术栈选择
- 性能与可靠性
- 安全考虑
- 可维护性与可扩展性

**产品功能设计重点关注**：
- 用户体验与使用场景
- 核心功能与优先级
- 产品边界
- 成功指标

**业务决策重点关注**：
- 业务价值与 ROI
- 风险与挑战
- 资源与约束
- 替代方案比较

#### 3.2 深入关键维度
对于每个选定的维度，使用 `references/questioning_patterns.md` 中的提问技巧：
- 使用 5W1H 框架探索细节
- 通过对比提问明确选择
- 用假设性问题探索边界
- 识别并挑战隐含假设

#### 3.3 螺旋式深入
不要一次性问完所有问题，而是：
1. 第一轮：宽度扫描，覆盖 3-5 个关键维度
2. 第二轮：根据回答，深入 2-3 个最重要的维度
3. 第三轮：补充遗漏的角度，再次验证关键决策

**参考案例**：查看 `references/examples.md` 中的成功案例，学习如何在不同场景下应用多维度思考。

### Phase 4: 方案完善（2-3 轮对话）

**目标**：形成具体、可执行的方案

#### 4.1 方案探索
- "还有其他方式可以达到同样的目标吗？"
- "这个方案的优势是什么？可能面临哪些挑战？"
- "类似的问题别人是怎么解决的？"

#### 4.2 细节确认
- "具体要如何实现这个功能？"
- "涉及哪些系统或模块？"
- "数据从哪里来，到哪里去？"
- "异常情况如何处理？"

#### 4.3 风险识别
- "最担心哪个环节出问题？"
- "如果失败了，最可能是什么原因？"
- "有什么备选方案？"

### Phase 5: 总结确认（1 轮对话）

**目标**：结构化输出，确保理解准确

#### 5.1 结构化总结
根据问题类型，整理成清晰的文档格式：

**技术方案**：
```markdown
## 问题背景
[核心问题、目标用户、当前痛点]

## 核心需求
[功能列表、优先级、范围边界]

## 技术方案
[架构设计、技术栈、关键模块]

## 实施计划
[分阶段交付、时间节点、人员分工]

## 风险与对策
[主要风险、应对措施、备选方案]
```

**产品功能**：
```markdown
## 功能概述
[核心价值、目标用户、使用场景]

## 功能详情
[功能列表、交互流程、边界条件]

## 用户体验
[关键路径、异常处理、性能要求]

## 成功指标
[量化指标、验收标准]

## 后续计划
[迭代规划、扩展方向]
```

**业务决策**：
```markdown
## 决策背景
[问题定义、决策目标、相关方]

## 方案对比
[候选方案、优缺点分析、评估维度]

## 推荐方案
[选择依据、预期收益、实施路径]

## 风险评估
[潜在风险、影响程度、缓解措施]

## 下一步行动
[具体行动项、责任人、时间节点]
```

#### 5.2 确认理解
- "让我总结一下我们讨论的内容：[总结要点]。我的理解准确吗？"
- "还有什么需要补充或调整的？"
- "你觉得哪些地方还不够清晰？"

## 高级技巧

### 1. 识别需要深挖的信号

当出现以下情况时，需要进一步提问：

**模糊表述**：
- 用户说："做一个系统"、"优化一下"、"改进性能"
- → 追问具体含义和量化标准

**隐含假设**：
- 用户说："肯定要用微服务"、"必须是实时的"
- → 验证假设是否合理，探索替代方案

**矛盾之处**：
- 用户说："要快速上线" + "功能要非常完善"
- → 澄清优先级和可接受的权衡

**缺失信息**：
- 用户只说了 what，没说 why 和 who
- → 补充背景、目标和受众

### 2. 提问的节奏控制

**避免信息过载**：
```
❌ 不好：一次提出 10 个问题
✅ 良好：每次 2-3 个相关问题，等用户回答后再继续
```

**逐步聚焦**：
```
Round 1: "你想做什么？" → "一个电商平台"
Round 2: "核心功能有哪些？" → "商品展示、购物车、支付"
Round 3: "支付环节最关心什么？" → "安全性和成功率"
Round 4: "支付安全具体考虑哪些方面？" → "加密、防重放、风控"
```

**灵活调整**：
- 如果用户回答简短，增加引导性问题
- 如果用户回答详细，减少提问，快速进入下一阶段
- 如果用户表现出疲劳，总结当前进展，询问是否继续

### 3. 苏格拉底式引导

不要直接给出答案，而是通过提问引导用户自己思考：

```
❌ 直接告知："你应该用微服务架构"

✅ 引导思考：
"你的系统预计有多少用户？各个模块的变更频率如何？
如果用单体架构，会遇到什么问题？
微服务架构能解决这些问题吗，会带来什么新的复杂度？"
```

### 4. 平衡深度与广度

**宽度优先**（初期探索）：
- 快速扫描所有重要维度
- 识别 2-3 个最关键的点

**深度优先**（关键点突破）：
- 对关键点进行纵向深挖
- 直到细节足够清晰可执行

**螺旋式深入**（推荐）：
- 多轮迭代，每轮都覆盖宽度和深度
- 第一轮：全局扫描
- 第二轮：关键点深入
- 第三轮：补充遗漏
- 第四轮：验证一致性

## 注意事项

### 1. 保持对话自然
- 不要机械地按照清单提问
- 根据用户的回答灵活调整提问顺序
- 用自然的对话语气，而不是面试官口吻

### 2. 尊重用户的知识水平
- 技术术语适度，根据用户背景调整
- 不要假设用户不懂，也不要过度解释
- 如果用户表现出专业性，快速进入深度讨论

### 3. 及时总结和确认
- 每个阶段结束时，简要总结已确认的信息
- 避免重复提问已经明确的内容
- 如果理解有偏差，及时修正

### 4. 关注用户的情绪和节奏
- 如果用户急于得到答案，加快节奏
- 如果用户享受探索过程，深入讨论
- 如果用户表现出困惑，用例子和类比解释

### 5. 知道何时结束
当满足以下条件时，可以结束头脑风暴：
- [ ] 问题定义清晰，没有歧义
- [ ] 目标和成功标准明确
- [ ] 范围和边界清晰
- [ ] 约束条件已识别
- [ ] 关键决策点已确认
- [ ] 潜在风险已讨论
- [ ] 实施路径基本清晰

## 示例对话片段

### 场景：用户想做一个系统（初期阶段）

**用户**："我想做一个任务管理系统"

**AI（初始理解）**：
"任务管理系统有很多种类型，能具体说说吗？
1. 这是给团队用的协作工具，还是个人用的待办事项？
2. 目前用什么方式管理任务？遇到了什么问题？"

**用户**："给团队用的，现在用 Excel 很乱，找不到任务状态"

**AI（范围界定）**：
"明白了。那团队大概多少人？主要需要哪些功能？
- 创建和分配任务
- 任务状态跟踪
- 进度可视化
- 协作讨论
- 其他？"

**用户**："10 个人左右，主要是前三个功能，讨论直接用微信就行"

**AI（多维度探索 - 选择关键维度）**：
"好的，接下来我想了解几个关键点：

**用户体验方面**：
任务分配和状态更新是最高频的操作吧？你希望操作有多简单？

**技术方面**：
团队有技术背景吗？是希望用现成的工具（如 Jira、Trello），还是定制开发？"

[继续对话...]

### 场景：性能优化（深度探索阶段）

**用户**："系统最近变慢了"

**AI（问题定位）**：
"「变慢」具体是指：
- 页面加载慢？
- API 响应慢？
- 某个特定功能慢？"

**用户**："订单列表页加载很慢"

**AI（深入细节）**：
"订单列表页，我想确认几个细节：
1. 加载大概多慢？（几秒钟？）
2. 以前也这样吗，还是最近才出现的？
3. 数据量有增长吗？"

**用户**："要等 5-6 秒，以前 1 秒就好了。最近订单从几万涨到几十万"

**AI（根因分析）**：
"数据量增长导致的性能问题，很常见。那：
1. 有看过数据库慢查询日志吗？
2. 列表查询有做索引优化吗？
3. 有用缓存吗？"

[继续深入，最终形成优化方案...]
