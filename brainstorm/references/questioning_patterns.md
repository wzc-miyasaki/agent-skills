# 提问模式和技巧

本文档提供有效的提问策略，帮助通过迭代式对话澄清和完善用户的想法。

## 核心原则

### 1. 渐进式澄清
从宏观到微观，从模糊到具体，逐步缩小问题范围。

### 2. 苏格拉底式提问
通过提问引导用户自己发现问题的本质，而不是直接给出答案。

### 3. 多角度验证
从不同维度反复确认，确保理解的准确性。

### 4. 挑战假设
识别并质疑隐含的假设，避免基于错误前提展开。

## 提问框架

### Stage 1: 问题定义（Problem Definition）

**目标**：理解用户真正想要解决的问题

```
核心问题探索：
- "你想要解决的核心问题是什么？"
- "为什么这个问题值得解决？"
- "这个问题给谁带来了困扰？"

现状分析：
- "目前的情况是怎样的？"
- "现有的解决方案有什么不足？"
- "如果不解决会有什么影响？"

目标明确化：
- "理想的结果是什么样的？"
- "如何判断问题已经被解决？"
- "有哪些具体的成功指标？"
```

### Stage 2: 范围界定（Scope Definition）

**目标**：明确边界，避免范围蔓延

```
范围确认：
- "这个方案需要覆盖哪些场景？"
- "有哪些场景明确不包括在内？"
- "MVP 和完整方案的区别是什么？"

优先级排序：
- "如果只能实现一个功能，会是什么？"
- "哪些是必须有的，哪些是可选的？"
- "前三个优先级最高的需求是什么？"

约束条件：
- "有什么时间、预算、资源上的限制？"
- "有哪些技术或业务上的约束？"
- "有什么不能改变的前提条件？"
```

### Stage 3: 方案探索（Solution Exploration）

**目标**：发散思维，探索多种可能性

```
替代方案：
- "还有其他方式可以达到同样的目标吗？"
- "类似的问题别人是怎么解决的？"
- "如果没有现有的限制，你会怎么做？"

权衡分析：
- "这个方案的优势是什么？"
- "可能会面临哪些挑战？"
- "哪些因素会影响方案的选择？"

风险识别：
- "最担心哪个环节出问题？"
- "如果失败了，最可能是什么原因？"
- "有什么备选方案？"
```

### Stage 4: 细节完善（Detail Refinement）

**目标**：深入关键细节，确保可执行

```
实现细节：
- "具体要如何实现这个功能？"
- "涉及哪些系统或模块？"
- "需要哪些外部依赖？"

数据和流程：
- "数据从哪里来，到哪里去？"
- "完整的流程是怎样的？"
- "异常情况如何处理？"

验证和测试：
- "如何验证方案是否有效？"
- "需要测试哪些场景？"
- "如何确保质量？"
```

## 高效提问技巧

### 1. 5W1H 框架
- **What**（什么）：要做什么？
- **Why**（为什么）：为什么要做？
- **Who**（谁）：为谁做？谁来做？
- **When**（何时）：什么时候需要？
- **Where**（何地）：在什么场景/环境下？
- **How**（如何）：如何实现？

### 2. 金字塔原理
从结论开始，逐层深入：
```
Level 1: "你想做什么？" → "一个用户管理系统"
Level 2: "系统需要哪些核心功能？" → "注册、登录、权限管理"
Level 3: "注册功能具体需要支持什么？" → "邮箱注册、手机号注册、第三方登录"
Level 4: "第三方登录需要支持哪些平台？" → "微信、支付宝"
```

### 3. 对比提问法
通过对比帮助用户明确选择：
```
二选一：
- "你更看重性能还是开发速度？"
- "用户体验优先还是功能完整性优先？"

程度量化：
- "从 1-10 分，这个功能有多重要？"
- "能接受的最长响应时间是多少？"

边界探测：
- "如果用户量增加 10 倍，系统能支持吗？"
- "如果预算减半，哪些功能可以砍掉？"
```

### 4. 假设验证法
识别并验证隐含假设：
```
识别假设：
- "你提到要用微服务架构，是基于什么考虑？"
- "为什么选择这个技术栈？"

挑战假设：
- "如果不用微服务，会有什么问题？"
- "用单体架构能满足需求吗？"

替代假设：
- "如果从零开始，你还会做同样的选择吗？"
- "有没有更简单的方案？"
```

### 5. 用户故事法
通过具体场景理解需求：
```
场景描述：
- "能描述一个典型用户的使用场景吗？"
- "用户从开始到结束会经历哪些步骤？"

异常场景：
- "如果用户输入了错误的信息会怎样？"
- "网络断了怎么办？"
- "并发访问会不会有问题？"
```

## 提问顺序策略

### 宽度优先（广度探索）
适用于：初始阶段，需要全面了解
```
第一轮：快速扫描所有维度
- 目标是什么？
- 用户是谁？
- 技术方案有想法吗？
- 有什么限制？

第二轮：根据回答聚焦重点维度
```

### 深度优先（纵向深挖）
适用于：某个维度特别重要或复杂
```
第一层：系统架构
第二层：后端设计
第三层：API 设计
第四层：认证机制
第五层：JWT 实现细节
```

### 螺旋式深入（推荐）
结合宽度和深度，反复迭代：
```
第一轮：宽度扫描，了解全貌
第二轮：识别 2-3 个关键点，适度深入
第三轮：补充遗漏的维度
第四轮：再次深入核心问题
第五轮：验证整体一致性
```

## 提问的节奏控制

### 单次提问量
- **初期探索**：1-3 个开放问题
- **深入阶段**：1-2 个聚焦问题
- **验证阶段**：2-4 个确认问题

### 避免信息过载
```
❌ 不好的方式：
"这个系统的目标用户是谁？有多少用户？用户的使用场景是什么？用户的痛点是什么？用户对性能的要求如何？用户需要什么样的界面？"

✅ 好的方式：
"这个系统的目标用户是谁？他们目前遇到什么问题？"
（等用户回答后再问下一批问题）
```

### 逐步聚焦
```
Round 1: "你想做什么？" → "一个电商平台"
Round 2: "电商平台的核心功能有哪些？" → "商品展示、购物车、支付"
Round 3: "支付环节最关心什么？" → "安全性和成功率"
Round 4: "支付安全具体需要考虑哪些方面？" → "数据加密、防重放、风控"
```

## 常见场景的提问模板

### 场景 1：技术方案设计
```
1. 要解决什么问题？（问题定义）
2. 预期的用户量和数据量？（规模评估）
3. 对性能和可用性的要求？（非功能需求）
4. 现有技术栈是什么？（技术约束）
5. 团队对哪些技术更熟悉？（团队能力）
6. 有时间和预算限制吗？（资源约束）
```

### 场景 2：功能需求澄清
```
1. 这个功能要解决用户的什么问题？（价值定位）
2. 用户会在什么场景下使用？（使用场景）
3. 有参考的产品或竞品吗？（对标分析）
4. 必须有哪些功能？哪些可以后续再加？（MVP 定义）
5. 如何衡量这个功能是成功的？（成功指标）
```

### 场景 3：日常问题解决
```
1. 遇到的具体问题是什么？（现象描述）
2. 期望达到什么结果？（目标明确）
3. 尝试过哪些方法？（已有尝试）
4. 有什么限制或特殊要求？（约束条件）
5. 可以接受哪些权衡？（灵活度）
```

### 场景 4：方案比较选择
```
1. 有哪几个候选方案？（方案列举）
2. 每个方案的优缺点是什么？（对比分析）
3. 最看重哪些因素？（决策标准）
4. 长期和短期的考虑分别是什么？（时间维度）
5. 团队的偏好和能力如何？（落地可行性）
```

## 识别需要深挖的信号

### 模糊表述
用户说："做一个系统"、"优化一下"、"改进性能"
→ 需要追问具体含义

### 隐含假设
用户说："肯定要用微服务"、"必须是实时的"
→ 需要验证假设是否合理

### 矛盾之处
用户说："要快速上线"+"功能要非常完善"
→ 需要澄清优先级

### 缺失信息
用户只说了 what，没说 why
→ 需要补充背景和目标

### 过度自信
用户说："这个很简单，一周就能做完"
→ 需要深入了解复杂度

## 结束头脑风暴的信号

### 明确性检查清单
- [ ] 问题定义清晰，没有歧义
- [ ] 目标和成功标准明确
- [ ] 范围和边界清晰
- [ ] 约束条件已识别
- [ ] 关键决策点已确认
- [ ] 潜在风险已讨论
- [ ] 实施路径基本清晰

### 最终确认
总结全部讨论要点，让用户确认理解是否准确：
```
"让我总结一下我们讨论的内容：
1. 你想要...
2. 核心需求是...
3. 技术方案考虑...
4. 主要约束包括...
我的理解准确吗？还有什么需要补充的？"
```
