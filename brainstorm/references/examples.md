# 头脑风暴成功案例

本文档展示多个成功的头脑风暴对话案例，展示如何通过迭代式提问将模糊想法澄清为具体方案。

## 案例 1：技术方案设计 - 用户管理系统

### 初始需求（模糊）
**用户**："我想做一个用户管理系统"

### 头脑风暴过程

**Round 1 - 问题定义**
```
AI: "能具体说说这个系统要解决什么问题吗？是为了管理哪类用户？"

用户: "是为公司内部的 SaaS 产品做用户管理，包括多个子系统"

AI: "明白了。那目前的痛点是什么？为什么需要新的用户管理系统？"

用户: "现在每个子系统都有自己的用户表，维护很麻烦，而且用户要记住多个账号密码"
```

**Round 2 - 范围界定**
```
AI: "所以核心需求是统一用户中心和单点登录对吧？那具体需要支持哪些功能呢？
     1. 用户注册登录
     2. 单点登录（SSO）
     3. 权限管理
     4. 其他？"

用户: "主要是前三个，权限这块可能要支持不同子系统的不同权限"

AI: "关于权限管理，你们需要：
     - 基于角色的权限控制（RBAC）
     - 还是基于资源的权限控制（RBAC）
     - 或者两者结合？"

用户: "RBAC 应该够用了，不需要太复杂"
```

**Round 3 - 技术约束**
```
AI: "了解了需求，接下来问几个技术方面的问题：
     1. 现有的子系统用什么技术栈开发的？
     2. 预计有多少用户量？
     3. 团队对哪些技术更熟悉？"

用户: "子系统都是 Spring Boot，用户量大概几千人吧，团队主要用 Java"

AI: "明白了。那关于单点登录，有考虑过具体的实现方案吗？
     - OAuth 2.0 + JWT
     - CAS（Central Authentication Service）
     - SAML
     还是其他方案？"

用户: "听说过 JWT，这个比较简单吧？"
```

**Round 4 - 方案深入**
```
AI: "JWT 确实是个好选择。不过我想确认几个关键点：
     1. Token 的有效期你们希望设置多长？
     2. 需要支持「记住我」功能吗？
     3. Token 刷新机制如何设计？
     4. 如果用户在一个子系统退出，其他系统要同时退出吗（单点登出）？"

用户: "有效期 2 小时吧，需要记住我，刷新机制...这个怎么做比较好？单点登出需要的"

AI: "好的，关于刷新机制，常见有两种方案：
     方案 1：Access Token（短期）+ Refresh Token（长期）
     方案 2：滑动过期时间（每次请求延长有效期）

     方案 1 更安全但实现复杂一点，方案 2 简单但安全性稍弱。考虑到是内部系统，你们倾向哪个？"

用户: "方案 1 吧，安全更重要"
```

**Round 5 - 数据库设计**
```
AI: "那数据库这块，需要存储这些核心信息：
     - 用户表（user）：用户基本信息
     - 角色表（role）：角色定义
     - 权限表（permission）：权限定义
     - 用户角色关联表（user_role）
     - 角色权限关联表（role_permission）

     另外，Refresh Token 需要持久化存储。这个设计可以吗？"

用户: "看起来可以，不过权限表是存每个子系统的权限吗？"

AI: "对，权限表需要设计成支持多系统的结构，建议加个字段：
     - system_id：标识属于哪个子系统
     - resource：资源标识（如 /api/users）
     - action：操作类型（如 read, write, delete）

     这样每个子系统可以定义自己的权限，但都在统一的权限中心管理。"

用户: "好的，这样清楚了"
```

### 最终输出（具体方案）
```
【用户管理系统方案总结】

## 核心需求
- 统一用户中心，替代各子系统独立的用户管理
- 单点登录（SSO），一次登录全系统可用
- 基于角色的权限控制（RBAC）
- 单点登出

## 技术方案
- 后端：Spring Boot + Spring Security
- 认证：OAuth 2.0 + JWT
- Token 策略：Access Token（2小时）+ Refresh Token（7天）
- 数据库：MySQL
- 缓存：Redis（存储 Token 黑名单和会话信息）

## 核心表结构
1. user（用户表）
2. role（角色表）
3. permission（权限表，包含 system_id）
4. user_role（用户角色关联）
5. role_permission（角色权限关联）
6. refresh_token（刷新令牌表）

## 主要流程
1. 登录流程：验证 → 生成 Access Token & Refresh Token → 返回客户端
2. 认证流程：请求携带 Access Token → 验证有效性 → 放行/拒绝
3. 刷新流程：Access Token 过期 → 用 Refresh Token 换取新 Access Token
4. 登出流程：Token 加入黑名单（Redis）→ 通知各子系统清理会话

## 用户量与性能
- 预计用户量：几千人（内部系统）
- 并发预估：较低
- 缓存策略：热点用户权限信息缓存到 Redis

## 后续扩展
- Phase 1：基础登录认证 + RBAC
- Phase 2：第三方登录（企业微信、钉钉）
- Phase 3：审计日志
```

## 案例 2：产品功能设计 - 文件分享功能

### 初始需求（模糊）
**用户**："想加个文件分享的功能"

### 头脑风暴过程

**Round 1 - 场景探索**
```
AI: "文件分享功能有很多种实现方式，能说说具体的使用场景吗？
     比如：谁会分享文件给谁？在什么情况下需要分享？"

用户: "就是用户上传了文件后，可以分享给别人看"

AI: "明白了。那「别人」是指：
     1. 系统内的其他用户
     2. 任何人（包括没有账号的）
     3. 还是都要支持？"

用户: "都要支持吧，有些文件想分享给客户看，客户可能没账号"
```

**Round 2 - 功能边界**
```
AI: "好的，那让我确认几个关键功能点：

     权限控制：
     - 需要设置查看权限吗（谁能看）？
     - 需要设置有效期吗（分享链接多久失效）？
     - 需要设置访问密码吗？

     操作权限：
     - 只能查看，还是也能下载？
     - 能编辑或评论吗？"

用户: "权限、有效期、密码都要。下载肯定要，编辑不需要，评论可以有"

AI: "关于有效期，你期望的默认值和可选范围是多少？
     比如：默认 7 天，可选 1天/7天/30天/永久？"

用户: "默认 7 天，永久有点危险吧..."

AI: "有道理。那最长支持多久？30天？90天？还是需要永久的选项但加上管理员审批？"

用户: "最长 90 天就够了"
```

**Round 3 - 用户体验**
```
AI: "接下来聊聊用户体验：

     分享方式：
     1. 生成链接，复制分享
     2. 直接填写对方邮箱，系统发送
     3. 两者都支持？

     你觉得哪种更符合用户习惯？"

用户: "生成链接吧,更灵活,用户想发给谁就发给谁"

AI: "那如果用户想批量分享多个文件呢？
     - 每个文件单独生成链接
     - 还是可以打包成一个分享链接？"

用户: "打包更方便，但单个文件也要能分享"

AI: "明白。关于接收方的体验：
     - 打开链接后直接看到文件，还是需要先预览？
     - 支持哪些文件格式的在线预览？（图片、PDF、Office 文档？）
     - 如果不支持预览，是否直接提供下载？"

用户: "最好能预览，图片和 PDF 肯定要支持，Office 文档...这个实现起来复杂吗？"
```

**Round 4 - 技术实现**
```
AI: "Office 文档预览有几个方案：
     1. 转换成 PDF 再预览（需要服务端转换）
     2. 使用第三方预览服务（如 Office Online、Google Docs Viewer）
     3. 前端库预览（如 mammoth.js 预览 Word）

     考虑到成本和复杂度：
     - 方案 1：服务端压力大，需要安装 LibreOffice 等工具
     - 方案 2：依赖外部服务，可能有隐私问题
     - 方案 3：预览效果一般，但轻量

     MVP 阶段建议先支持图片和 PDF，Office 文档提供下载，后续再考虑预览。可以吗？"

用户: "可以，先保证核心功能"
```

**Round 5 - 安全考虑**
```
AI: "最后聊聊安全性，这个很重要：

     防止滥用：
     - 需要限制分享次数吗（比如每天最多创建 100 个分享）？
     - 需要限制单次分享的文件大小或数量吗？

     隐私保护：
     - 访问日志需要记录吗（谁在什么时候访问了）？
     - 分享者能看到访问记录吗？

     恶意内容：
     - 需要检测病毒或敏感内容吗？"

用户: "限制分享次数有必要，防止滥用。访问日志要记录，分享者应该能看到。病毒检测...现在可以先不做吗？"

AI: "可以的，MVP 先不做病毒扫描。那限制次数设置为多少合适？
     考虑正常用户使用频率和防止滥用的平衡。"

用户: "每天 50 次够了，正常用户不会分享这么多"
```

### 最终输出（具体 PRD）
```
【文件分享功能 PRD】

## 功能概述
用户可以将已上传的文件生成分享链接，分享给系统内外的任何人查看和下载。

## 核心功能

### 1. 创建分享
- 支持单个文件分享
- 支持多个文件打包分享
- 配置项：
  - 有效期：1天/7天(默认)/30天/90天
  - 访问密码：可选
  - 权限设置：仅查看 / 允许下载（默认）

### 2. 访问分享
- 打开链接后：
  - 如有密码，先验证密码
  - 验证有效期
  - 展示文件列表（多文件）或文件详情（单文件）
- 在线预览：
  - 图片：支持（jpg, png, gif, webp）
  - PDF：支持
  - Office 文档：MVP 阶段不支持，提供下载
  - 其他格式：提供下载
- 下载：支持单个下载和批量下载（zip 打包）

### 3. 分享管理
- 分享列表：查看我创建的所有分享
- 分享详情：查看访问记录（IP、时间、操作）
- 取消分享：提前使链接失效
- 分享统计：查看次数、下载次数

## 限制规则
- 每个用户每天最多创建 50 个分享
- 单次分享最多 100 个文件
- 单次分享总大小不超过 2GB
- 分享链接最长有效期 90 天

## 技术方案
- 分享链接生成：短链接（8 位随机码）
- 链接格式：https://domain.com/s/{shareCode}
- 密码加密：bcrypt
- 访问日志：异步写入数据库
- 文件预览：前端直接渲染（图片、PDF）
- 打包下载：后端临时生成 zip，定时清理

## 数据库设计
```sql
-- 分享表
CREATE TABLE file_share (
  id BIGINT PRIMARY KEY,
  share_code VARCHAR(8) UNIQUE,
  user_id BIGINT,
  expire_time DATETIME,
  access_password VARCHAR(255),
  allow_download BOOLEAN DEFAULT TRUE,
  visit_count INT DEFAULT 0,
  download_count INT DEFAULT 0,
  created_at DATETIME,
  status TINYINT -- 0:失效 1:正常
);

-- 分享文件关联表
CREATE TABLE share_file (
  id BIGINT PRIMARY KEY,
  share_id BIGINT,
  file_id BIGINT
);

-- 访问日志表
CREATE TABLE share_access_log (
  id BIGINT PRIMARY KEY,
  share_id BIGINT,
  ip VARCHAR(50),
  action VARCHAR(20), -- view, download
  created_at DATETIME
);
```

## UI 交互流程
1. 用户选择文件 → 点击「分享」按钮
2. 弹窗配置分享参数（有效期、密码等）
3. 生成链接，显示在弹窗内，支持一键复制
4. 接收方打开链接 → 输入密码（如有）→ 预览/下载文件

## 未来扩展
- Phase 2：支持 Office 文档在线预览
- Phase 2：分享给指定用户（需要登录才能访问）
- Phase 3：协作编辑
- Phase 3：邮件通知（创建分享时自动发送邮件）
```

## 案例 3：日常问题解决 - 性能优化

### 初始需求（模糊）
**用户**："我们的系统最近变慢了，想优化一下性能"

### 头脑风暴过程

**Round 1 - 问题定位**
```
AI: "「变慢了」具体是指哪方面慢了？
     - 页面加载慢？
     - API 响应慢？
     - 数据库查询慢？
     - 还是整体都慢？"

用户: "主要是某几个页面加载很慢，要等好几秒"

AI: "是哪几个页面？这些页面有什么共同特点吗？
     比如都涉及大量数据查询、复杂计算或者第三方接口调用？"

用户: "主要是订单列表页和用户详情页，订单列表加载要 5-6 秒，用户详情页也要 3-4 秒"
```

**Round 2 - 现状分析**
```
AI: "好的，先了解一下现状：
     1. 这两个页面以前也这么慢吗，还是最近才出现的？
     2. 如果是最近才慢的，数据量或用户量有明显增长吗？
     3. 有做过性能监控或日志分析吗？"

用户: "以前还好,最近一个月开始明显变慢。数据量确实增长了不少，订单从几万涨到了几十万。监控...有装阿里云的应用监控"

AI: "能看看监控数据吗？主要关注：
     - 这两个接口的响应时间分布
     - 慢查询 SQL
     - 服务器 CPU 和内存使用率

     或者你能描述一下监控显示的情况吗？"

用户: "监控显示订单列表接口 P99 响应时间 8 秒，有几条 SQL 查询时间超过 2 秒"
```

**Round 3 - 深入分析**
```
AI: "看来主要是 SQL 查询慢。那我们具体看看这两个页面：

     订单列表页：
     - 一次加载多少条数据？有分页吗？
     - 列表展示哪些字段？
     - 有关联查询吗（比如订单关联用户、商品等）？
     - 有复杂的筛选条件吗（时间范围、状态、关键词搜索等）？"

用户: "每页 20 条，有分页。列表要显示订单号、用户名、商品名、金额、状态、创建时间。有关联查询，订单表要关联用户表和订单商品表。筛选条件有时间范围、状态、用户名搜索"

AI: "明白了，这是典型的关联查询性能问题。再问几个细节：
     1. 订单表、用户表、订单商品表的数据量分别是多少？
     2. 这些表有建索引吗？特别是关联字段和筛选字段？
     3. SQL 语句用的是 JOIN 还是分步查询？"

用户: "订单表 50 万，用户表 10 万，订单商品表 100 万。索引...好像只有主键索引，其他没怎么建。SQL 用的是 LEFT JOIN 查询的"
```

**Round 4 - 方案探讨**
```
AI: "问题找到了！50 万订单做多表 JOIN 且缺少索引，性能肯定很差。

     我建议几个优化方向，按优先级排序：

     【优先级 P0 - 立即优化】
     1. 添加索引：
        - 订单表：status, user_id, created_at
        - 订单商品表：order_id
        - 组合索引：(status, created_at) 用于列表筛选

     【优先级 P1 - 近期优化】
     2. 优化查询策略：
        - 减少 JOIN，改为分步查询 + 内存组装
        - 用户名搜索改为先查用户 ID，再查订单

     3. 引入缓存：
        - 热门查询条件的结果缓存（Redis）
        - 用户信息缓存，避免每次查用户表

     【优先级 P2 - 长期优化】
     4. 数据库分库分表（数据量继续增长后考虑）

     你觉得这个方案如何？有什么担心的点吗？"

用户: "加索引听起来挺简单的，这个能直接上生产吗？会不会影响现有业务？"

AI: "加索引需要注意几点：
     1. 在生产环境加索引会锁表（MySQL 5.6 及以上支持 Online DDL，影响较小）
     2. 建议在业务低峰期操作（比如凌晨）
     3. 先在测试环境验证索引效果
     4. 索引会增加写入成本，但对你们的场景影响不大（订单列表查询多，写入少）

     具体步骤：
     1. 今晚在测试环境加索引，测试查询性能提升
     2. 如果效果明显（预计能提升 10 倍以上），明晚凌晨在生产环境操作
     3. 操作前做好数据库备份

     这样可以吗？"

用户: "可以，那用户详情页的问题呢？"
```

**Round 5 - 另一个问题**
```
AI: "用户详情页的优化思路类似。先问几个问题：
     1. 用户详情页展示哪些信息？
     2. 有聚合查询吗（比如统计用户的订单数、消费总额等）？
     3. 响应慢主要是哪个部分？"

用户: "展示用户基本信息、最近订单、消费统计（总订单数、总金额、最近下单时间）。消费统计是实时算的，每次都要 COUNT 和 SUM"

AI: "找到问题了！实时聚合查询在大数据量下很慢。优化方案：

     方案 1：冗余字段（推荐）
     - 在用户表增加字段：order_count, total_amount, last_order_time
     - 每次下单时更新这些字段（事务内或异步）
     - 详情页直接读取，无需实时计算
     - 优点：查询极快；缺点：数据可能有短暂延迟

     方案 2：缓存计算结果
     - 首次访问时计算并缓存到 Redis（TTL 5 分钟）
     - 后续访问直接读缓存
     - 下单后清除该用户的缓存
     - 优点：实现简单；缺点：首次访问仍然慢

     方案 3：定时统计
     - 每小时跑一次批量统计任务
     - 结果存到统计表
     - 优点：对实时性要求不高的场景适用；缺点：数据延迟较大

     考虑到你们的场景，推荐方案 1。你觉得呢？"

用户: "方案 1 好，但是会不会有并发问题？比如多个订单同时更新这些字段"

AI: "好问题！确实需要考虑并发控制。解决方案：

     1. 使用数据库的原子操作：
        UPDATE user SET
          order_count = order_count + 1,
          total_amount = total_amount + #{amount},
          last_order_time = NOW()
        WHERE user_id = #{userId}

     2. 或者使用乐观锁（version 字段）

     3. 如果对实时性要求不高，可以异步更新（MQ 削峰填谷）

     推荐方案 1 + 3 组合：下单时发 MQ 消息，消费者异步更新统计字段，用原子操作避免并发问题。"

用户: "明白了，这样确实更稳妥"
```
